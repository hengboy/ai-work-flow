# 深化

如何在已知依赖的情况下，安全地深化一组浅层模块。假设您熟悉 [SKILL.md](SKILL.md) 中的术语——**模块**、**接口**、**接缝**、**适配器**。

## 依赖分类

在评估一个待深化的候选模块时，需要对其依赖进行分类。依赖的类别决定了深化后的模块如何通过其接缝进行测试。

### 1. 进程内

纯计算逻辑，内存中的状态，无 I/O 操作。此类依赖始终可以深化——将相关模块合并，并直接通过新接口进行测试。无需适配器。

### 2. 可本地替换

指那些存在本地测试替身的依赖（例如，使用 PGLite 替代 Postgres，或使用内存文件系统）。如果存在这种替身，则此类依赖可以深化。深化后的模块将在测试套件中使用该替身运行进行测试。此时的接缝是内部的，模块的外部接口上无需端口。

### 3. 远程但可控（端口与适配器）

指跨越网络边界的自有服务（微服务、内部 API 等）。在接缝处定义一个 **端口**（接口）。深化后的模块负责业务逻辑；传输层则以 **适配器** 的形式注入。测试时使用内存适配器，生产环境则使用 HTTP/gRPC/队列适配器。

推荐表述：_“在接缝处定义一个端口，在生产环境中实现 HTTP 适配器，在测试中使用内存适配器，这样即使逻辑部署在多个服务之间，核心业务逻辑仍然集中在一个深化后的模块中。”_

### 4. 真正的外部依赖（模拟）

指第三方服务（如 Stripe、Twilio 等）等不由您控制的依赖。深化后的模块将这些外部依赖作为注入的端口，测试时提供一个模拟适配器。

## 接缝规范

- **一个适配器意味着一个假想的接缝。两个适配器才构成真正的接缝。** 除非至少有两个适配器的理由（通常是生产环境和测试环境），否则不要引入端口。单适配器的接缝只是多余的间接层。
- **内部接缝与外部接缝。** 深化后的模块既可以拥有内部接缝（仅限于其实现内部，供其自身测试使用），也可以在其对外接口处设置外部接缝。切勿因为测试使用了内部接缝，就将其暴露到外部接口上。

## 测试策略：替换而非叠加

- 一旦深化后的模块拥有了自己的接口级测试，原有针对浅层模块的单元测试便不再适用——请予以删除。
- 在深化后的模块接口处编写新的测试。**接口即测试面**。
- 测试应关注通过接口观察到的结果，而非模块内部状态。
- 测试应能经受住内部重构的考验——它们描述的是行为，而非实现细节。如果实现发生变化时测试也需要修改，则说明测试超出了接口范围。
