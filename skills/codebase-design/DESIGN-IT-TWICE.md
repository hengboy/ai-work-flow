# 两次设计

当用户希望为选定的深化候选方案探索替代接口时，使用这种并行子代理模式。基于“两次设计”（Ousterhout）——你的第一个想法很可能不是最好的。

使用 [SKILL.md](SKILL.md) 中的术语——**模块**、**接口**、**边界**、**适配器**、**杠杆效应**。

## 流程

### 1. 框定问题空间

在启动子代理之前，为选定的候选方案撰写一份面向用户的背景说明：

- 新接口需要满足的约束条件
- 它所依赖的其他部分，以及这些依赖属于哪一类（参见 [DEEPENING.md](DEEPENING.md)）
- 一个粗略的示例代码草图，用于具体化这些约束——这并非提案，只是让约束更加直观的一种方式

将这份说明展示给用户，随后立即进入步骤 2。用户在阅读并思考的同时，子代理们会并行工作。

### 2. 启动子代理

使用 Agent 工具并行启动 3 个或更多子代理。每个子代理必须为待深化的模块设计出一个 **截然不同** 的接口。

为每个子代理提供一份独立的技术简报（包括文件路径、耦合细节、来自 [DEEPENING.md](DEEPENING.md) 的依赖类别，以及边界背后的内容）。该简报与步骤 1 中面向用户的背景说明无关。为每个代理设定不同的设计约束：

- 子代理 1： “尽量简化接口——目标是最多 1–3 个入口点。最大化每个入口点的杠杆效应。”
- 子代理 2： “追求最大灵活性——支持多种用例和扩展能力。”
- 子代理 3： “针对最常见的调用方进行优化——让默认场景变得极其简单。”
- 子代理 4（如适用）：“围绕端口与适配器设计，处理跨边界依赖。”

在简报中同时融入 [SKILL.md](SKILL.md) 和 CONTEXT.md 中的术语，使每个子代理都能以与架构语言及项目领域语言一致的方式命名相关概念。

每个子代理的输出包括：

1. 接口定义（类型、方法、参数——以及不变量、调用顺序、错误处理方式）
2. 调用示例，展示调用方如何使用该接口
3. 边界背后实现层隐藏的具体内容
4. 依赖策略与适配器（参见 [DEEPENING.md](DEEPENING.md)）
5. 各项权衡——哪些地方杠杆效应显著，哪些地方较弱

### 3. 展示与对比

依次展示各设计方案，让用户逐一理解，随后用文字对它们进行比较。从 **深度**（接口处的杠杆效应）、**局部性**（变更集中发生的范围）以及 **边界位置** 等方面进行对比。

对比结束后，给出你的建议：认为哪个设计最为优秀，并说明理由。如果不同设计中的某些元素可以很好地结合，可提出一种混合方案。务必表达明确的观点——用户需要的是有力的判断，而非简单的选项罗列。
