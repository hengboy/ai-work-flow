# 代码库设计

设计**深度模块**：在小规模接口背后封装大量行为，置于清晰的分界线上，并可通过该接口进行测试。无论何时设计或重构代码，都应遵循这套语言和原则。目标是为调用方提供杠杆效应，为维护者提供局部性，为所有人提供可测试性。

## 术语表

请严格按照这些术语使用——不要用“组件”、“服务”、“API”或“边界”来替代。保持语言的一致性才是关键。

**模块** — 任何具有接口和实现的东西。刻意做到与规模无关：可以是一个函数、类、包，也可以是跨越多个层级的部分。_避免使用_：单元、组件、服务。

**接口** — 调用者正确使用模块所需了解的一切：类型签名，以及不变量、顺序约束、错误模式、所需配置和性能特征。_避免使用_：API、签名（范围过窄——它们仅指类型层面的表面）。

**实现** — 模块内部的代码主体。与**适配器**不同：一个事物既可以是小型适配器搭配大型实现（如Postgres仓库），也可以是大型适配器搭配小型实现（如内存中的模拟）。当讨论分界线本身时，使用“适配器”；否则使用“实现”。

**深度** — 接口处的杠杆效应：调用者（或测试）每学习单位接口所能触发的行为量。当大量行为隐藏在小规模接口后时，模块被称为**深度模块**；当接口几乎与实现一样复杂时，则称为**浅层模块**。

**分界线** _(Michael Feathers)_ — 可以在不修改该位置的情况下改变行为的地方；即模块接口所在的位置。分界线的放置是独立于其背后内容的设计决策。_避免使用_：边界（被DDD的限界上下文过度占用）。

**适配器** — 在分界线上满足接口的具体实体。描述的是*角色*（它所处的位点），而非具体内容（内部是什么）。

**杠杆效应** — 调用者从深度中获得的收益：每学习单位接口就能获得更多能力。一次实现可在N个调用端和M个测试中得到回报。

**局部性** — 维护者从深度中获得的收益：变更、Bug、知识和验证都集中在一个地方，而不是扩散到各个调用端。一处修复，处处生效。

## 深度与浅层

**深度模块** = 小型接口 + 大量实现：

```
┌─────────────────────┐
│   Small Interface   │  ← Few methods, simple params
├─────────────────────┤
│                     │
│  Deep Implementation│  ← Complex logic hidden
│                     │
└─────────────────────┘
```

**浅层模块** = 大型接口 + 少量实现（应避免）：

```
┌─────────────────────────────────┐
│       Large Interface           │  ← Many methods, complex params
├─────────────────────────────────┤
│  Thin Implementation            │  ← Just passes through
└─────────────────────────────────┘
```

设计接口时，请问自己：

- 是否可以减少方法数量？
- 是否可以简化参数？
- 是否可以将更多复杂性隐藏在内部？

## 原则

- **深度是接口的属性，而非实现的属性。** 深度模块的内部可以由小型、可模拟、可替换的部件组成——只是它们不属于接口的一部分。模块可以拥有**内部分界线**（仅供其实现内部使用，供其自身测试使用），以及位于接口处的**外部分界线**。
- **删除测试。** 想象一下删除这个模块。如果复杂性随之消失，说明它只是个简单的传递层；如果复杂性在N个调用端重新出现，说明它确实发挥了作用。
- **接口就是测试面。** 调用者和测试会经过同一条分界线。如果想越过接口进行测试，那么这个模块可能形状不对。
- **一个适配器意味着假想的分界线，两个适配器则意味着真实的分界线。** 除非确实有东西跨分界线变化，否则不要引入分界线。

## 针对可测试性的设计

良好的接口让测试变得自然：

1. **接受依赖，不要创建它们。**

   ```typescript
   // 易于测试
   function processOrder(order, paymentGateway) {}

   // 难以测试
   function processOrder(order) {
     const gateway = new StripeGateway();
   }
   ```

2. **返回结果，不要产生副作用。**

   ```typescript
   // 易于测试
   function calculateDiscount(cart): Discount {}

   // 难以测试
   function applyDiscount(cart): void {
     cart.total -= discount;
   }
   ```

3. **尽量缩小表面积。** 方法越少，所需的测试就越少；参数越少，测试设置就越简单。

## 关系

- 一个**模块**恰好拥有一个**接口**（即它向调用者和测试呈现的表面）。
- **深度**是**模块**的一种属性，以其**接口**为基准衡量。
- **分界线**是**模块**的**接口**所在的位置。
- **适配器**位于**分界线**上，满足**接口**的要求。
- **深度**为调用者带来**杠杆效应**，为维护者带来**局部性**。

## 被拒绝的表述方式

- **将深度视为实现代码行数与接口代码行数的比值**（Ousterhout）：这种定义会鼓励人为增加实现部分的冗余。我们改用基于杠杆效应的深度定义。
- **将“接口”等同于TypeScript中的`interface`关键字或类的公有方法**：范围过于狭窄——这里的接口包括调用者必须知道的每一条信息。
- **“边界”**：被DDD的限界上下文概念过度占用。建议使用**分界线**或**接口**。
