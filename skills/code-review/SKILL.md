## 专职代理路由

本技能由 **Coordinator** 路由。Coordinator 只与用户交互、调度、等待和汇总，不直接访问工作区、运行 Shell、编辑文件或实施。将全库枚举、`glob`、`grep` 和代码地图交给 **File Explorer**；外部一手资料交给 **Researcher**；普通文档交给 **Document Maintainer**；计划、任务、ADR、交接和跟踪器文本交给 **Planning Writer**；源码、测试、配置、调试和提交交给 **Full Stack Coder**；稳定差异的双轴评审交给 **Code Reviewer**。除 File Explorer 外，所有本地角色只读取其交接的路径及直接依赖；写入角色串行执行。生成工作流后，以 `~/.config/ai-work-flow/routing.md` 为最终规则。下文的每个命令和第二人称指代均由相应受委派角色执行，绝不由 Coordinator 执行。

对 `HEAD` 和用户提供的固定点之间的差异进行双轴评审：

- **标准** — 代码是否符合本仓库的文档化编码规范？
- **规格** — 代码是否忠实地实现了原始问题 / PRD / 规格？

两个维度由 **Code Reviewer** 以 **Review Standards** 和 **Review Spec** 两个固定内部评审器并行运行，互不干扰各自上下文，随后由 Coordinator 汇总发现。

问题跟踪器应已提供给您——如果 `docs/agents/issue-tracker.md` 文件缺失，请运行 `/setup-matt-pocock-skills`。

## 流程

### 1. 确定固定点

无论用户指定为何种固定点——提交 SHA、分支名称、标签、`main`、`HEAD~5` 等——均需确认。若未指定，请向用户询问。

仅需捕获一次差异命令：`git diff <fixed-point>...HEAD`（使用三重省略号，比较的是合并基点）。同时记录提交列表：`git log <fixed-point>..HEAD --oneline`。

在继续之前，确认固定点有效（`git rev-parse <fixed-point>`），且差异非空。无效引用或空差异应在这一阶段直接报错，而非交由两个并行子代理处理。

### 2. 确定规格来源

按以下顺序查找原始规格：

1. 提交信息中的问题引用（如 `#123`、`Closes #45`、GitLab `!67` 等）——可通过 `docs/agents/issue-tracker.md` 中的工作流获取。
2. 用户作为参数传递的路径。
3. 匹配分支名或功能的 `docs/`、`specs/` 或 `.scratch/` 下的 PRD/规格文件。
4. 若仍未找到，请询问用户规格所在位置。若用户表示没有规格，则 **Spec** 子代理将跳过并报告“无可用规格”。

### 3. 确定标准来源

仓库中任何用于说明代码编写方式的文档，例如 `CODING_STANDARDS.md` 或 `CONTRIBUTING.md`。

除仓库文档外，**Standards** 轴始终包含以下 **气味基准** ——一套固定的 Fowler 代码异味（参见《重构》第 3 章），即使仓库未明确文档化编码规范也适用。其遵循两条规则：

- **仓库优先。** 文档化的仓库规范始终优先；若规范认可了基准会标记的某种做法，则无需报告该异味。
- **始终基于判断。** 每种异味均为带标签的启发式建议（如“可能的特征嫉妒”），并非硬性违规——且如同此处所有标准一样，工具已强制执行的部分可忽略。

每种异味的表述形式为 _表现_ → _修复方法_；将其与差异对比：

- **神秘名称** — 函数、变量或类型名称未能揭示其用途或所持内容。→ 重命名；若无法取到合理名称，则表明设计存在模糊之处。
- **重复代码** — 同一逻辑结构出现在变更中的多个代码块或文件中。→ 抽取共享部分，分别调用。
- **特征嫉妒** — 某方法过多访问其他对象的数据而非自身数据。→ 将该方法移至其“嫉妒”的数据所在对象。
- **数据团块** — 相同的几个字段或参数反复出现（类似一个亟待封装的类型）。→ 将其打包成一个新类型，统一传递。
- **原始类型情结** — 原始类型或字符串被用来代替本应拥有独立类型的领域概念。→ 为该概念创建一个小型专用类型。
- **重复的 switch 语句** — 同一类型的 `switch`/`if` 多次出现在变更中。→ 替换为多态，或使用一处映射表供双方共享。
- **霰弹式修改** — 一项逻辑变更导致差异中多处文件发生零散编辑。→ 将相关变更集中到一个模块中。
- **发散式修改** — 单个文件或模块因多种无关原因被修改。→ 分割，使每个模块仅因单一原因而变更。
- **臆测的通用性** — 为尚未出现的需求添加的抽象、参数或钩子。→ 删除之，逐步内联直至出现实际需求。
- **消息链** — 长串 `a.b().c().d()` 导航，调用方不应依赖于此。→ 将整个调用过程封装到首个对象上的一个方法中。
- **中间人** — 某类或函数几乎只负责转发调用。→ 删除之，直接调用真正目标。
- **拒绝继承** — 某子类或实现者忽视或覆盖了大部分继承内容。→ 放弃继承，改用组合。

### 4. 并行启动两个子代理

Code Reviewer 通过一条消息同时启动 Review Standards 和 Review Spec。除这两个固定内部评审器外，Reviewer 不得分派其他角色。

**Standards 子代理提示** — 包括：

- 完整的差异命令及提交列表。
- 第 3 步中找到的全部标准源文件，**以及第 3 步中完整粘贴的气味基准**——子代理无法访问其他途径。
- 简要说明：“请按文件/代码块逐项报告：(a) 差异中违反文档化标准的所有地方，注明标准出处（文件名称及具体条款）；(b) 发现的任何基准异味，注明其名称并引用相关代码块。区分硬性违规与判断性意见——文档化标准的违背可能是硬性违规，但基准异味始终属于判断性意见，且文档化的仓库标准优先于基准。工具已强制执行的部分可忽略。字数不超过 400 字。”

**Spec 子代理提示** — 包括：

- 差异命令及提交列表。
- 规格的路径或已获取的内容。
- 简要说明：“请报告：(a) 规格要求但缺失或未完全实现的功能；(b) 差异中未被要求的行为（范围蔓延）；(c) 表面上看似已实现、但实现方式存在问题的功能。每项发现均需引用规格原文。字数不超过 400 字。”

若规格缺失，则跳过 **Spec** 子代理，并在最终报告中注明。

### 5. 汇总

将两份报告分别置于 `## Standards` 和 `## Spec` 标题下，原样呈现或稍作清理。**切勿合并或重新排序**发现——两个维度刻意保持分离（参见“为何采用双轴”）。

最后以一行总结收尾：各维度的总发现数量，以及各自范围内最严重的问题（如有）。切勿跨维度评选优劣——这种重新排序正是分离机制旨在避免的。

## 为何采用双轴

一项变更可能在一个维度上通过，而在另一个维度上失败：

- 代码完全符合所有标准，却实现了错误的功能 → **Standards 通过，Spec fail。**
- 代码完全满足问题要求，却违反了项目约定 → **Spec 通过，Standards fail。**

分轴报告可防止一个维度掩盖另一个维度。
