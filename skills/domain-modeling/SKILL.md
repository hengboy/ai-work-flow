## 专职代理路由

本技能由 **Coordinator** 路由。Coordinator 只与用户交互、调度、等待和汇总，不直接访问工作区、运行 Shell、编辑文件或实施。将全库枚举、`glob`、`grep` 和代码地图交给 **File Explorer**；外部一手资料交给 **Researcher**；普通文档交给 **Document Maintainer**；计划、任务、ADR、交接和跟踪器文本交给 **Planning Writer**；源码、测试、配置、调试和提交交给 **Full-Stack Coder**；稳定差异的双轴评审交给 **Code Reviewer**。除 File Explorer 外，所有本地角色只读取其交接的路径及直接依赖；写入角色串行执行。生成工作流后，以 `.ai-work-flow/agents/routing.md` 为最终规则。下文的每个命令和第二人称指代均由相应受委派角色执行，绝不由 Coordinator 执行。

# 领域建模

在设计过程中主动构建并完善项目的领域模型。这是*主动*的实践——挑战现有术语、构思边界场景，并在概念清晰的瞬间立即将术语表和决策记录下来。（仅仅阅读 `CONTEXT.md` 来获取词汇并不属于本技能——那只是任何技能都可以完成的一条简单习惯。本技能适用于你正在改变模型的时候，而不仅仅是消费它。）

## 文件结构

大多数仓库只有一个上下文：

```
/
├── CONTEXT.md
├── docs/
│   └── adr/
│       ├── 0001-event-sourced-orders.md
│       └── 0002-postgres-for-write-model.md
└── src/
```

如果根目录下存在 `CONTEXT-MAP.md`，则该仓库包含多个上下文。该映射文件会指向每个上下文的具体位置：

```
/
├── CONTEXT-MAP.md
├── docs/
│   └── adr/                          ← system-wide decisions
├── src/
│   ├── ordering/
│   │   ├── CONTEXT.md
│   │   └── docs/adr/                 ← context-specific decisions
│   └── billing/
│       ├── CONTEXT.md
│       └── docs/adr/
```

按需创建文件——只有在有内容可写时才创建。如果没有 `CONTEXT.md`，在第一个术语确定时立即创建。如果没有 `docs/adr/`，在需要第一条 ADR 时再创建。

## 会话期间

### 对照术语表进行质疑

当用户使用的术语与 `CONTEXT.md` 中的现有定义冲突时，应立即指出。“你的术语表将‘取消’定义为 X，但你似乎指的是 Y——到底哪个才是正确的？”

### 澄清模糊表述

当用户使用含糊或含义多重的术语时，提出一个精确的规范术语。“你说的‘账户’是指 Customer 还是 User？这两者是不同的概念。”

### 讨论具体场景

在讨论领域关系时，通过具体场景对其进行压力测试。设想一些能够触及边界情况的场景，迫使用户对概念之间的边界做出更精确的界定。

### 与代码交叉核对

当用户说明某项功能的工作方式时，检查代码是否与其一致。如果发现矛盾，及时指出：“你的代码会取消整个订单，但你刚才说部分取消也是可能的——到底哪种说法正确？”

### 即时更新 `CONTEXT.md`

一旦某个术语的定义得到确认，就地更新 `CONTEXT.md`。不要积攒待处理事项，而应在问题出现时立即记录。请参考 [CONTEXT-FORMAT.md](./CONTEXT-FORMAT.md) 中的格式。

`CONTEXT.md` 应完全不含实现细节。切勿将其视为规格说明、临时草稿或存放实现决策的地方。它只是一个术语表，仅此而已。

### 谨慎提供 ADR

仅在同时满足以下三个条件时才建议创建 ADR：

1. **难以撤销**——日后更改决定的成本较高
2. **缺乏背景信息时令人意外**——未来的读者可能会疑惑“为什么当时要这样做？”
3. **基于真实权衡的结果**——确实存在多种备选方案，且你出于特定原因选择了其中之一

如果上述条件中任一不满足，则无需创建 ADR。请参考 [ADR-FORMAT.md](./ADR-FORMAT.md) 中的格式。
