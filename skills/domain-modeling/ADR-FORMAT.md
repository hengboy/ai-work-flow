# ADR 格式

ADR 存放在 `docs/adr/` 目录中，并使用连续编号：`0001-slug.md`、`0002-slug.md` 等。

仅在需要第一个 ADR 时，再创建 `docs/adr/` 目录。

## 模板

```md
# {Short title of the decision}

{1-3 sentences: what's the context, what did we decide, and why.}
```

就是这样。一个 ADR 可以只是一段文字。其价值在于记录*已经做出过某个决策*以及*为什么这么做*，而不是去填满各种章节。

## 可选章节

只有当这些内容确实能带来价值时才添加。大多数 ADR 并不需要它们。

- **状态** frontmatter（`proposed | accepted | deprecated | superseded by ADR-NNNN`）—— 当决策被重新审视时很有用
- **备选方案** —— 仅在需要记住被否决的替代方案时才添加
- **后果** —— 仅在需要指出一些不明显的下游影响时才添加

## 编号

扫描 `docs/adr/` 目录，找到现有的最大编号并加一。

## 何时撰写 ADR

以下三个条件必须同时满足：

1. **难以撤销** —— 后续改变主意的成本很高
2. **缺乏上下文时令人意外** —— 未来的读者看到代码会疑惑：“他们为什么会这样设计？”
3. **基于实际权衡的决策** —— 曾存在多种可行的替代方案，而你出于特定原因选择了其中之一

如果某个决策很容易撤销，那就不用写 ADR —— 随便改回来就行。如果这个决策并不出人意料，就不会有人产生疑问。如果没有真正的替代方案，那也无需记录，只需注明“我们选择了最显而易见的做法”。

### 哪些情况适合记录为 ADR

- **架构层面的设计选择**：“我们采用单体仓库。”“写入模型基于事件溯源，读取模型则通过投影写入 PostgreSQL。”
- **不同上下文之间的集成模式**：“订单模块与计费模块通过领域事件通信，而非同步 HTTP 调用。”
- **具有锁定效应的技术选型**：数据库、消息总线、认证服务提供商、部署目标等。并非所有库都适用，而是那些替换起来需要花费四分之一时间的组件。
- **边界与范围的界定**：“客户数据由客户上下文负责；其他上下文仅通过 ID 引用。”明确的否定同样重要，甚至比肯定更为关键。
- **刻意偏离常规路径的做法**：“我们使用原生 SQL 而非 ORM，因为 X。”凡是会让合理预期的读者认为相反做法才是正确的场景，都应该记录下来，以免下一位工程师误以为这是个可以“修复”的问题。
- **代码中无法直接体现的约束条件**：“由于合规要求，我们不能使用 AWS。”“响应时间必须低于 200 毫秒，这是合作伙伴 API 协议的要求。”
- **当拒绝某些方案的原因并不明显时**：如果你曾考虑过 GraphQL 但因某些微妙原因选择了 REST，就应当记录下来——否则六个月后可能还会有人再次提出使用 GraphQL。
