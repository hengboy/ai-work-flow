## 专职代理路由

本技能由 **Coordinator** 路由。Coordinator 只与用户交互、调度、等待和汇总，不直接访问工作区、运行 Shell、编辑文件或实施。将全库枚举、`glob`、`grep` 和代码地图交给 **File Explorer**；外部一手资料交给 **Researcher**；普通文档交给 **Document Maintainer**；计划、任务、ADR、交接和跟踪器文本交给 **Planning Writer**；源码、测试、配置、调试和提交交给 **Full-Stack Coder**；稳定差异的双轴评审交给 **Code Reviewer**。除 File Explorer 外，所有本地角色只读取其交接的路径及直接依赖；写入角色串行执行。生成工作流后，以 `~/.config/ai-work-flow/routing.md` 为最终规则。下文的每个命令和第二人称指代均由相应受委派角色执行，绝不由 Coordinator 执行。

# 改善代码库架构

揭示架构层面的摩擦，并提出**深化机会**——将浅层模块重构为深层模块。目标是提升可测试性和 AI 导航能力。

该命令基于项目的领域模型并建立在共同的设计词汇之上：

- 运行 `/codebase-design` 技能，获取架构相关的词汇（**模块**、**接口**、**深度**、**边界**、**适配器**、**杠杆效应**、**局部性**）及其原则（删除测试、“接口即测试面”、“一个适配器 = 假设的边界，两个 = 真实的边界”）。在所有建议中严格使用这些术语——不要偏离到“组件”、“服务”、“API”或“边界”等表述。
- `CONTEXT.md` 中的领域语言为良好的边界命名；`docs/adr/` 中的 ADR 记录了本命令不应重新讨论的决策。

## 流程

### 1. 探索

首先阅读项目领域的术语表（`CONTEXT.md`）以及您即将触及区域的相关 ADR。

随后将代码库遍历委派给 **File Explorer**。不要调用原生 `Explore` 代理；File Explorer 应以有机的方式探索，并记录下遇到的摩擦点：

- 在哪些地方，理解一个概念需要在多个小模块之间来回切换？
- 哪些模块是**浅层的**——其接口几乎与实现一样复杂？
- 是否有纯粹函数仅为了可测试性而被提取，但真正的 Bug 却隐藏在其调用方式中（缺乏**局部性**）？
- 哪些紧密耦合的模块越过了它们的边界？
- 代码库的哪些部分未被测试，或者通过当前接口难以测试？

对任何疑似浅层的部分应用**删除测试**：如果将其删除，是否会集中复杂度，还是仅仅将其转移？如果答案是“会集中复杂度”，那就是您要寻找的信号。

### 2. 将候选方案以 HTML 报告形式呈现

将一份自包含的 HTML 文件写入操作系统的临时目录，确保不会有任何内容进入代码仓库。从 `$TMPDIR` 获取临时目录路径，若不存在则回退到 `/tmp`（Windows 下为 `%TEMP%`），并将文件保存为 `<tmpdir>/architecture-review-<timestamp>.html`，以便每次运行都生成一个新文件。随后为用户打开该文件——Linux 下使用 `xdg-open <path>`，macOS 下使用 `open <path>`，Windows 下使用 `start <path>`——并向用户提供其绝对路径。

报告采用**通过 CDN 引入的 Tailwind CSS**进行布局和样式设计，并使用**通过 CDN 引入的 Mermaid**绘制图表，用于以图形/流程/序列清晰表达结构。将 Mermaid 图与手绘的 CSS/SVG 视觉效果相结合——当关系呈现为图状（如调用图、依赖关系、序列图）时使用 Mermaid，而当需要更具编辑性的表达（如质量分布图、剖面图、折叠动画）时则使用手绘的 div 和 SVG。每个候选方案都配有**优化前后的可视化对比**。务必注重视觉呈现。

对于每个候选方案，展示一张卡片，内容包括：

- **涉及的文件**——哪些文件或模块受到影响
- **问题描述**——当前架构为何导致摩擦
- **解决方案**——用通俗易懂的英语描述将如何改进
- **收益**——从局部性和杠杆效应的角度解释，并说明测试将如何改善
- **优化前后示意图**——并排展示，定制绘制，直观呈现浅层与深化后的差异
- **推荐强度**——分为 `Strong`、`Worth exploring`、`Speculative` 三档，并以标签形式显示

报告末尾附上**重点推荐**部分：说明您认为应优先处理的候选方案及其理由。

**请在描述领域相关概念时使用 `CONTEXT.md` 中的词汇，在描述架构相关概念时使用 `/codebase-design` 中的词汇。** 如果 `CONTEXT.md` 定义了“订单”，请称其为“订单录入模块”，而不是“FooBarHandler”或“订单服务”。

**ADR 冲突**：如果某个候选方案与现有 ADR 冲突，仅当摩擦确实严重到值得重新审视该 ADR 时才予以提示。并在卡片中明确标注（例如添加警示说明：“与 ADR-0007 冲突——但值得重新开启讨论，因为……”）。切勿列出 ADR 禁止的所有理论重构方案。

完整 HTML 模板、图表模式及样式指南参见 [HTML-REPORT.md](HTML-REPORT.md)。

**请勿在此阶段提出接口设计。** 文件生成后，请询问用户：“您希望先探讨哪一个方案？”

### 3. 深化讨论循环

用户选定候选方案后，运行 `/grilling` 技能，与其一同梳理设计方案树——包括约束条件、依赖关系、深化后模块的形态、边界背后的内容以及哪些测试能够保留。

随着决策逐步明晰，副作用会同步发生——运行 `/domain-modeling` 技能，实时更新领域模型：

- **是否要为深化后的模块命名一个 `CONTEXT.md` 中未提及的概念？** 请将该术语补充至 `CONTEXT.md`。若文件尚不存在，则按需创建。
- **在讨论过程中是否需要进一步明确某个模糊术语？** 请当场更新 `CONTEXT.md`。
- **若用户因重要理由拒绝该候选方案？** 可提供 ADR 形式记录：“需要我将其写入 ADR，以便未来的架构评审不再重复提出此建议吗？”仅在该理由确实可能在未来避免类似建议时才提出，对于短暂性理由（如“目前不值得”）或显而易见的理由则无需记录。
- **是否希望探索深化后模块的其他接口方案？** 请运行 `/codebase-design` 技能，并利用其“双轨设计”子代理模式。
