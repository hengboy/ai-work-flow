本技能由 **Coordinator** 路由。Coordinator 只与用户交互、调度、等待和汇总，不直接访问工作区、运行 Shell、编辑文件或实施。将全库枚举、`glob`、`grep` 和代码地图交给 **File Explorer**；外部一手资料交给 **Researcher**；普通文档交给 **Document Maintainer**；计划、任务、ADR、交接和跟踪器文本交给 **Planning Writer**；源码、测试、配置、调试和提交交给 **Full-Stack Coder**；稳定差异的双轴评审交给 **Code Reviewer**。除 File Explorer 外，所有本地角色只读取其交接的路径及直接依赖；写入角色串行执行。生成工作流后，以 `~/.config/ai-work-flow/routing.md` 为最终规则。下文的每个命令和第二人称指代均由相应受委派角色执行，绝不由 Coordinator 执行。

# 测试驱动开发

TDD 是一个红 → 绿 的循环过程。本技能指南旨在确保这个循环能够产出值得保留的测试：包括什么是好的测试、测试应该放在哪里、常见的反模式以及循环中的规则。每个部分都适用于每一次迭代——请在进入循环之前和过程中查阅，而不是事后回顾。

在探索代码库时，请阅读 `CONTEXT.md`（如果存在），以确保测试名称和接口词汇与项目的领域语言保持一致，并尊重你所触及区域的 ADR 文档。

## 什么是好的测试

测试通过公共接口验证行为，而非实现细节。即使代码完全改变，测试也不应受到影响。一个好的测试读起来像一份规范——例如，“用户可以使用有效购物车完成结账”，明确地告诉你系统具备了什么能力——并且能够在重构中幸存下来，因为它不依赖于内部结构。

更多示例请参阅 [tests.md](tests.md)，关于模拟的指导原则请参阅 [mocking.md](mocking.md)。

## 接缝——测试的位置

**接缝**是你进行测试的公开边界：即观察行为而不深入内部的接口。测试应当位于接缝处，绝不能针对内部实现。

**仅在事先约定好的接缝处进行测试。** 在编写任何测试之前，先列出要测试的接缝，并与用户确认。未经确认的接缝不得进行测试。你不可能测试所有内容——提前确定接缝，是为了将测试重点放在关键路径和复杂逻辑上，而不是每一个边缘场景。

提问：“什么是公共接口？我们应该测试哪些接缝？”

## 常见的反模式

- **与实现耦合**——模拟内部协作对象、测试私有方法，或通过旁路渠道进行验证（例如直接查询数据库而非使用接口）。其特征是：当你进行重构但行为并未改变时，测试就会失败。
- **同义反复**——断言以代码同样的方式重新计算期望值（如 `expect(add(a, b)).toBe(a + b)`，手动生成的快照与代码生成方式相同，或将常量与其自身比较），因此测试只是按构造通过，永远无法与代码产生分歧。期望值必须来自独立的权威来源——已知正确的字面值、经过验证的例子或规范本身。
- **水平切片**——先写完所有测试，再补全所有实现。这种批量测试验证的是**想象中的**行为：你测试的是事物的**形状**，而非面向用户的功能；测试对真实的变化变得不敏感；并且你在尚未理解实现细节之前就锁定了测试结构。建议改为**垂直切片**——一次测试 → 一次实现 → 重复进行，每项测试都是一颗**追踪弹**，根据上一周期的经验不断调整。

## 循环的规则

- **先红后绿。** 先编写一个会失败的测试，然后只添加刚好能让它通过的代码。不要预先设想未来的测试，也不要加入任何推测性功能。
- **一次处理一个切片。** 每次迭代只关注一个接缝、一个测试和一个最简实现。
- **重构不属于循环的一部分。** 它属于评审阶段（参见 `code-review` 技能），而不是红 → 绿的实现循环。
