## 专职代理路由

本技能由 **Coordinator** 路由。Coordinator 只与用户交互、调度、等待和汇总，不直接访问工作区、运行 Shell、编辑文件或实施。将全库枚举、`glob`、`grep` 和代码地图交给 **File Explorer**；外部一手资料交给 **Researcher**；普通文档交给 **Document Maintainer**；计划、任务、ADR、交接和跟踪器文本交给 **Planning Writer**；源码、测试、配置、调试和提交交给 **Full-Stack Coder**；稳定差异的双轴评审交给 **Code Reviewer**。除 File Explorer 外，所有本地角色只读取其交接的路径及直接依赖；写入角色串行执行。生成工作流后，以 `~/.config/ai-work-flow/routing.md` 为最终规则。下文的每个命令和第二人称指代均由相应受委派角色执行，绝不由 Coordinator 执行。

# 分类

将项目的问题跟踪器中的问题通过一个小型的分类角色状态机进行流转。

如果该仓库将外部拉取请求视为请求入口（参见 issue-tracker 配置），则分类也会覆盖这些请求：**PR 是带有代码的问题**——相同的分类角色、相同的状态、相同的流程，仅在下文中标注了“针对 PR”的一些差异。根据问题跟踪器配置，将纯 `#42` 解析为问题或 PR。

在分类过程中，任何发布到问题跟踪器的评论或问题 **必须** 以以下免责声明开头：

```
> *This was generated by AI during triage.*
```

## 角色

两个 **类别** 角色：

- `bug` — 某处存在故障
- `enhancement` — 新功能或改进

五个 **状态** 角色：

- `needs-triage` — 维护者需要评估
- `needs-info` — 等待报告者提供更多信息
- `ready-for-agent` — 完全明确，可供离线代理处理
- `ready-for-human` — 需要人工实现
- `wontfix` — 不会采取行动

对于 PR，相同的状态是基于附带的代码来判断的：`ready-for-agent` 表示已附加简报，代理应根据差异继续下一步；`ready-for-human` 表示已准备好由人工合并。

每个经过分类的问题应同时具有一个类别角色和一个状态角色。如果状态角色发生冲突，请先标记并征得维护者的同意，再采取进一步行动。

这些是规范的角色名称——实际在问题跟踪器中使用的标签字符串可能有所不同。映射关系应已提供给您；若未提供，请运行 `/setup-matt-pocock-skills`。

状态转换规则：未加标签的问题通常首先进入 `needs-triage` 状态；随后可转至 `needs-info`、`ready-for-agent`、`ready-for-human` 或 `wontfix`。`needs-info` 在报告者回复后会返回到 `needs-triage`。维护者可在任何时候进行干预——如发现异常状态转换，请先标记并询问后再继续。

## 调用方式

维护者调用 `/triage` 并用自然语言描述其需求。解析请求并执行相应操作。示例：

- “给我看看所有需要我关注的内容”
- “我们来看看 #42 吧”（问题或 PR）
- “把 #42 移动到 ready-for-agent 状态”
- “有哪些可以交给代理处理的？”

## 显示需要关注的问题

查询问题跟踪器并按时间顺序列出三个分组：

1. **未加标签** — 从未分类过。
2. **`needs-triage`** — 正在评估中。
3. **`needs-info` 且自上次分类记录以来有报告者活动** — 需要重新评估。

当 PR 属于范围时，将外部 PR 列入上述分组，并在每行前标注 `[PR]` 或 `[issue]`。发现阶段仅显示 *外部* PR（谁算作外部由问题跟踪器配置决定）——协作者正在进行的 PR 不属于分类工作。此筛选仅用于发现阶段；显式指定的 PR 均需分类，无论作者是谁。

显示每项的数量及一行摘要，供维护者选择。

## 对特定问题或 PR 进行分类

1. **收集上下文。** 阅读完整的问题或 PR（正文、评论、标签、作者、日期；对于 PR，还包括差异）。解析之前的分类记录，避免重复询问已解决的问题。结合项目的领域术语表探索代码库，遵守相关领域的 ADR。对代码库进行两项检查：(a) **冗余性** — 按领域概念搜索是否存在与请求行为类似的现有实现（不局限于请求措辞），并记录搜索位置。若找到，则视为已实现的 `wontfix`（步骤 5）。(b) **先前被拒绝** — 阅读 `.out-of-scope/*.md`，找出与本次请求相似的条目。

2. **提出建议。** 向维护者说明您的类别和状态建议及其理由，并附上与请求相关的简要代码库概述——包括是否已有实现。等待指示。

3. **验证主张。** 在深入讨论之前，先确认主张是否成立。对于 bug，按照报告者的步骤重现问题。对于 PR，确认差异确实实现了其宣称的功能——检出代码，运行相关测试或命令。汇报结果：确认（并给出代码路径）、失败，或细节不足（强烈提示 `needs-info`）。经确认的验证能使代理简报更具说服力。

4. **深入讨论（如有必要）。** 若请求尚需完善，联合使用 `/grilling` 和 `/domain-modeling` 技能——逐个问题将其细化，同步更新领域术语和 `CONTEXT.md`/ADRs。

5. **应用处理结果：**
   - `ready-for-agent` — 发布一份代理简报评论（参见 [AGENT-BRIEF.md](AGENT-BRIEF.md)）。
   - `ready-for-human` — 结构与代理简报类似，但需注明为何无法委托处理（涉及判断、外部访问、设计决策、手动测试等）。
   - `needs-info` — 发布分类记录（模板见下文）。
   - `wontfix` — 关闭问题，并根据原因添加注释：
     - **已实现** — 更改已在代码库中存在。指出具体位置；**切勿**写入 `.out-of-scope/`（该知识库专用于 *被拒绝的* 请求，而非已实现的功能）。
     - **被拒绝（bug）** — 礼貌解释后关闭。
     - **被拒绝（增强）** — 写入 `.out-of-scope/`，并在评论中链接至该条目，然后关闭（参见 [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md)）。
   - `needs-triage` — 直接应用该角色。若有部分进展，可选添加注释。

## 快速状态覆盖

若维护者要求“将 #42 移动到 ready-for-agent 状态”，请遵从并直接应用该角色。确认即将执行的操作（角色变更、评论、关闭），然后实施。无需深入讨论。若在未进行讨论的情况下直接移动到 `ready-for-agent`，请询问是否需要撰写代理简报。

## `needs-info` 模板

```markdown
## Triage Notes

**What we've established so far:**

- point 1
- point 2

**What we still need from you (@reporter):**

- question 1
- question 2
```

将讨论过程中已明确的内容归入“目前已明确的内容”，以免遗漏。问题须具体且可操作，而非“请提供更多信息”。

## 继续上一阶段的工作

若问题或 PR 上已有之前的分类记录，请阅读并检查报告者是否已回复未决问题，在继续前呈现更新后的状况。避免重复询问已解决的问题。
