## 专职代理路由

本技能由 **Coordinator** 路由。Coordinator 只与用户交互、调度、等待和汇总，不直接访问工作区、运行 Shell、编辑文件或实施。将全库枚举、`glob`、`grep` 和代码地图交给 **File Explorer**；外部一手资料交给 **Researcher**；普通文档交给 **Document Maintainer**；计划、任务、ADR、交接和跟踪器文本交给 **Planning Writer**；源码、测试、配置、调试和提交交给 **Full-Stack Coder**；稳定差异的双轴评审交给 **Code Reviewer**。除 File Explorer 外，所有本地角色只读取其交接的路径及直接依赖；写入角色串行执行。生成工作流后，以 `.ai-work-flow/agents/routing.md` 为最终规则。下文的每个命令和第二人称指代均由相应受委派角色执行，绝不由 Coordinator 执行。

# 诊断 Bug

针对棘手 Bug 的一种方法论。只有在有明确理由的情况下才能跳过某些阶段。

在探索代码库时，请阅读 `CONTEXT.md`（如果存在），以获得相关模块的清晰概念模型，并查看你所涉及区域的 ADR 文档。

## 第一阶段 — 构建反馈回路

**这是核心技能。** 其余部分都只是机械操作。如果你能为该 Bug 提供一个**紧密的**通过/失败信号——即在这个 Bug 上会变红的信号——那么你就能找到问题根源；二分查找、假设验证和代码插桩等手段都是围绕它展开的。如果没有这样的信号，再多盯着代码看也无济于事。

请在此阶段投入更多精力。**要积极主动，勇于创新，绝不轻言放弃。**

### 构建反馈回路的方法 — 按照大致顺序尝试

1. 在能够触及该 Bug 的任何接口处编写**失败测试**——单元测试、集成测试或端到端测试。
2. 对正在运行的开发服务器执行**Curl 或 HTTP 脚本**。
3. 使用固定输入调用**CLI 命令行工具**，并将标准输出与已知正确的快照进行比对。
4. 编写**无头浏览器脚本**（Playwright 或 Puppeteer）——驱动 UI 并对 DOM、控制台或网络请求进行断言。
5. **重放捕获的跟踪日志。** 将真实的网络请求、有效载荷或事件日志保存到磁盘中，然后在隔离的代码路径中重新执行。
6. **搭建临时测试环境。** 启动系统的最小子集（单个服务，模拟依赖项），仅通过一次函数调用来触发该 Bug 的代码路径。
7. **属性测试或模糊测试循环。** 如果该 Bug 是“有时输出错误”，则运行 1000 次随机输入，寻找故障模式。
8. **二分排查测试环境。** 如果该 Bug 出现在两个已知状态之间（提交记录、数据集或版本），则自动化“启动到状态 X，检查，重复”的流程，以便你可以使用 `git bisect run` 进行二分排查。
9. **差异对比循环。** 将同一输入分别通过旧版本和新版本（或两种配置）运行，并比较输出结果差异。
10. **人机协作 Bash 脚本。** 最后手段。如果必须由人工点击，则使用 `scripts/hitl-loop.template.sh` 来引导他们，使整个过程仍保持结构化。捕获的输出会反馈给你。

构建好合适的反馈回路，Bug 就已经解决了 90%。

### 优化反馈回路

将反馈回路视为一个产品。一旦有了初步的回路，就应着手对其进行**优化**：

- 能否让它运行得更快？（缓存初始化步骤，跳过无关的初始化，缩小测试范围）
- 能否让信号更明显？（针对具体症状进行断言，而不是简单地判断“未崩溃”）
- 能否让它更加确定性？（固定时间，设置随机数种子，隔离文件系统，冻结网络）

一个需要 30 秒且不稳定的工作循环，其实和没有回路差不多；而一个只需 2 秒且确定性的回路才是真正的高效工具，堪称调试利器。

### 非确定性 Bug

目标并不是完全重现，而是提高**重现率**。将触发条件循环执行 100 次，同时并行运行，增加压力，缩小时间窗口，插入延迟等待。一个 50% 稳定性的 Bug 是可以调试的；而 1% 稳定性的 Bug 则无法调试——持续提升重现率，直到可调试为止。

### 当确实无法构建反馈回路时

此时应停下来并明确说明情况。列出你已尝试过的所有方法，并向用户请求：(a) 获取能够重现该 Bug 的环境访问权限；(b) 请求提供捕获的样本（HAR 文件、日志转储、核心转储或带时间戳的屏幕录像）；(c) 请求允许添加临时的生产环境监控代码。**切勿在没有反馈回路的情况下直接进入假设阶段。**

### 完成标准 — 一个能变红的紧密回路

第一阶段完成的标志是拥有一条**紧密且能变红**的回路：你能说出**一条命令**——脚本路径、测试调用或 Curl 命令——并且你**至少已经运行过一次**（粘贴该命令及其输出），同时满足以下条件：

- [ ] **能变红**——它能够真正触发该 Bug 的代码路径，并断言用户的**确切症状**，因此在 Bug 存在时会变红，修复后会变绿。不能只是“不会报错”，而必须能够**捕捉到这个特定的 Bug**。
- [ ] **确定性**——每次运行结果一致（对于非确定性 Bug，则需确保较高的重现率，如上所述）。
- [ ] **快速**——几秒钟内完成，而不是几分钟。
- [ ] **无人值守**——你可以让它无人干预地运行；若有人参与，也只能通过 `scripts/hitl-loop.template.sh` 来实现。

如果你发现自己在还没有这条命令之前就开始读代码推测原因，**请立即停止——直接跳到假设阶段正是本技能要避免的陷阱。** 没有能变红的命令，就不能进入第二阶段。

## 第二阶段 — 重现并精简

运行你的反馈回路，观察其变红的过程——Bug 出现了。

确认以下几点：

- [ ] 回路确实产生了用户描述的**故障模式**——而不是附近出现的其他故障。找错了 Bug 就会修错地方。
- [ ] 该故障能够在多次运行中稳定重现（对于非确定性 Bug，则需达到足够高的重现率，以便进行调试）。
- [ ] 你已准确记录下该 Bug 的**确切症状**（错误信息、错误输出或响应过慢），以便后续阶段能够验证修复是否真正解决问题。

### 精简

一旦回路变红，将重现场景缩减至**仍然能变红的最小化方案**。逐次移除输入、调用者、配置、数据和步骤，每移除一项后重新运行回路——只保留对故障起关键作用的部分。

为什么要这样做：最小化的重现场景能够缩小第三阶段的假设空间（减少需要怀疑的变量数量），并在第五阶段成为可靠的回归测试。

当**剩余的每个元素都对故障不可或缺**时，即可认为完成。移除其中任意一个元素都会使回路变绿。

在成功**重现并精简**之前，不得继续下一步。

## 第三阶段 — 提出假设

在测试任何假设之前，先生成**3–5 个排序后的假设**。如果只提出一个假设，往往会停留在第一个看似合理的念头上。

每个假设都必须是**可证伪的**：明确指出其预测结果。

> 格式：“如果 <X> 是原因，那么改变 <Y> 将使 Bug 消失 / 改变 <Z> 会使 Bug 更严重。”

如果无法明确预测结果，则该假设只是主观猜测——要么丢弃，要么进一步细化。

**在测试前将排序后的假设列表展示给用户。** 用户通常具备领域知识，能够立刻调整优先级（“我们刚部署了 #3 的变更”），或者知道哪些假设已被排除。这是一个低成本的检查点，能节省大量时间。如果用户暂时无法参与，也不必因此停滞，可先按自己的判断进行排序。

## 第四阶段 — 插桩调试

每次探查都必须对应第三阶段中的某个具体预测。**每次只改变一个变量。**

工具选择偏好：

1. 如果环境支持，优先使用**调试器或 REPL 检查**。一个断点胜过十条日志。
2. 在能够区分不同假设的边界处添加**有针对性的日志**。
3. 切忌“把所有东西都记录下来再 grep”。

**为每条调试日志加上唯一的前缀标签**，例如 `[DEBUG-a4f2]`。这样在清理时只需一次 grep 即可删除所有调试日志。未加标签的日志可以保留，加了标签的则全部清除。

**性能分支。** 对于性能回归问题，日志往往难以准确反映真实情况。此时应先建立基准测量（计时框架、`performance.now()`、性能分析器或查询计划），然后进行二分排查。先测量，后修复。

## 第五阶段 — 修复并回归测试

**在修复之前先编写回归测试**——但前提是存在一个**合适的测试接口**。

合适的接口是指该测试能够模拟**实际的 Bug 行为模式**，并覆盖其在调用位置的真实表现。如果唯一可用的接口过于浅显（例如，单个调用者的单元测试，而该 Bug 需要多个调用者才能触发，或者单元测试无法复现引发 Bug 的完整链路），那么基于此接口的回归测试只会带来虚假的信心。

**如果不存在合适的接口，这本身就是一项发现。** 记录下来。这意味着当前的代码架构设计阻碍了对该 Bug 的彻底解决。将这一问题标记出来，留待下一阶段处理。

如果存在合适的接口：

1. 将精简后的重现场景转化为该接口上的失败测试。
2. 观察其失败。
3. 应用修复方案。
4. 观察其通过。
5. 再次运行第一阶段的反馈回路，验证原始（未精简）场景是否正常。

## 第六阶段 — 清理与事后分析

在宣布任务完成之前，必须完成以下事项：

- [ ] 原始重现场景不再重现该 Bug（再次运行第一阶段的回路）
- [ ] 回归测试通过（或明确记录没有合适接口的情况）
- [ ] 所有 `[DEBUG-...]` 标记的调试代码均已移除（通过 grep 查找前缀）
- [ ] 临时原型已删除（或移至明确标注的调试区域）
- [ ] 在提交或 PR 信息中明确指出最终被证实正确的假设——以便下一位调试人员从中学习。

**最后问自己：是什么导致了这个 Bug？** 如果答案涉及架构层面的改进（缺乏良好的测试接口、调用关系混乱、隐藏耦合），则将具体问题移交 `/improve-codebase-architecture` 技能处理。务必在修复完成后提出改进建议，而不是在开始前就讨论架构问题——此时你已掌握更多相关信息。
