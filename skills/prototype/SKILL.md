## 专职代理路由

本技能由 **Coordinator** 路由。Coordinator 只与用户交互、调度、等待和汇总，不直接访问工作区、运行 Shell、编辑文件或实施。将全库枚举、`glob`、`grep` 和代码地图交给 **File Explorer**；外部一手资料交给 **Researcher**；普通文档交给 **Document Maintainer**；计划、任务、ADR、交接和跟踪器文本交给 **Planning Writer**；源码、测试、配置、调试和提交交给 **Full Stack Coder**；稳定差异的双轴评审交给 **Code Reviewer**。除 File Explorer 外，所有本地角色只读取其交接的路径及直接依赖；写入角色串行执行。生成工作流后，以 `~/.config/ai-work-flow/routing.md` 为最终规则。下文的每个命令和第二人称指代均由相应受委派角色执行，绝不由 Coordinator 执行。

# 原型

原型是**用来回答某个问题的一次性代码**。问题决定了它的形态。

## 选择分支

确定要回答的问题——可以从用户的提示、周围的代码中推断，或者在用户在线时直接询问：

- **“这段逻辑/状态模型是否合理？”** → [LOGIC.md](LOGIC.md)。构建一个小型的交互式终端应用，将状态机推进到那些在纸上难以推理的状态。
- **“这个应该是什么样子？”** → [UI.md](UI.md)。为同一个路由生成几种截然不同的 UI 变体，通过 URL 查询参数和浮动底部栏进行切换。

这两个分支会产出非常不同的产物——选错分支会导致整个原型浪费。如果问题确实模糊不清且无法联系到用户，应默认选择与周围代码更匹配的分支（后端模块→逻辑；页面或组件→UI），并在原型顶部明确说明这一假设。

## 两条分支共用的规则

1. **从第一天起就是一次性的，并明确标注**。将原型代码放置在靠近实际使用位置的地方（紧邻它所模拟的模块或页面），以便上下文清晰可见——但命名时要让普通读者一眼看出这是原型而非生产代码。对于一次性 UI 路由，遵循项目已有的路由约定，不要新建顶层结构。
2. **一条命令即可运行**。无论项目现有的任务运行器支持什么方式——`pnpm <name>`、`python <path>`、`bun <path>` 等——用户都必须无需思考就能启动它。
3. **默认不持久化**。状态仅保存在内存中。持久化是原型要验证的内容，而不是它自身依赖的东西。如果问题明确涉及数据库，则连接一个临时数据库，或使用一个带有明显“PROTOTYPE — 清空我”标识的本地文件。
4. **跳过打磨环节**。无需测试，错误处理只需保证原型能够**运行**即可，也不引入任何抽象。重点在于快速获取所需信息。
5. **暴露状态**。每次操作（逻辑）之后，或每次切换变体（UI）时，打印或渲染出完整的相关状态，让用户清楚地看到发生了哪些变化。
6. **完成后记录下来**。将经过验证的决策融入正式代码，然后将原型本身作为**一手资料**保留：提交到一个独立的废弃分支，远离主分支，并在实现相关的 issue 中留下指向该分支的上下文链接。同时将答案——即最终结论及其所解决的问题——记录在 issue 或 commit 中。主分支只保留经过验证的决策。
