# UI 原型

在一个路由上生成**几种截然不同的 UI 变体**，可通过浮动的底部栏进行切换。用户在浏览器中翻看这些变体，选择一个（或从每个变体中借鉴一些元素），然后丢弃其余部分。

如果问题涉及的是逻辑/状态而非外观——那就选错了分支。请参考 [LOGIC.md](LOGIC.md)。

## 何时适合采用这种形式

- “这个页面应该是什么样子？”
- “我想在提交之前先看看这个仪表盘的几个备选方案。”
- “为设置页面尝试一种不同的布局。”
- 任何用户原本会在脑海中花一天时间在三个模糊的线框图之间做选择的情况。

## 两种子形态——强烈推荐子形态 A

当 UI 原型**与应用的其他部分紧密衔接**时，评估起来会容易得多——真实的页头、侧边栏、数据和布局密度。单独存在的临时路由就像一个真空：每个变体在孤立状态下看起来都还不错。只要有一个合理的现有页面可以承载这些变体，就默认使用子形态 A。只有当原型确实没有合适的归属页面时，才考虑子形态 B。

### 子形态 A——对现有页面的调整（首选）

该路由已经存在。变体**在同一路由上渲染**，通过 `?variant=` URL 查询参数来控制显示。现有的数据获取、参数和认证机制保持不变，仅渲染部分会切换。这是默认选项；除非有特殊原因，否则应优先选择此方式。

如果原型是针对尚未有页面但*自然会归属于某个页面的内容*（如仪表盘的新板块、设置页面的新卡片、现有流程中的新步骤），这也属于子形态 A。将变体挂载到宿主页面内部即可。

### 子形态 B——新建页面（最后手段）

仅当所要原型化的内容确实没有现有页面可归属时才使用这种方式——例如全新的顶级界面，或无法嵌入到任何合理位置的流程。

按照项目已有的路由规范创建一个**临时路由**——不要引入新的顶级结构。命名时应明确表明其原型性质（例如在路径或文件名中加入 `prototype`）。同样使用 `?variant=` 参数模式。

在决定采用子形态 B 之前，请先确认：真的没有任何现有页面可以容纳它吗？空洞的路由会掩盖那些在已有内容填充的情况下会暴露的设计问题。

在两种子形态中，浮动底部栏的实现完全相同。

## 流程

### 1. 明确问题并确定数量 N

默认提供**3 个变体**。超过 5 个就会失去“截然不同”的意义，变成干扰——因此上限定为 5 个。

在原型所在位置或文件顶部注释中用一句话写下计划：

> “设置页面的三种变体，通过 `?variant=` 参数切换，位于现有的 `/settings` 路由上。”

无论用户是否持反对意见，这一步都适用。

### 2. 设计截然不同的变体

分别草拟每个变体。每个变体需满足以下要求：

- 符合页面的用途及所能访问的数据。
- 遵循项目的组件库或样式体系（TailwindCSS、shadcn、MUI、纯 CSS 等）。
- 提供清晰的导出组件名称，如 `VariantA`、`VariantB`、`VariantC`。

变体必须在**结构上显著不同**——布局、信息层级、主要交互方式等均应有所区别，而不仅仅是颜色上的变化。仅仅稍作调整的卡片网格排列并不构成 UI 原型，更像是壁纸。如果两个草案过于相似，应重新设计其中一个，并明确指示“不得使用卡片网格”。

### 3. 将变体整合到一起

在路由上创建一个统一的切换组件：

```tsx
// pseudo-code — adapt to the project's framework
const variant = searchParams.get('variant') ?? 'A';
return (
  <>
    {variant === 'A' && <VariantA {...data} />}
    {variant === 'B' && <VariantB {...data} />}
    {variant === 'C' && <VariantC {...data} />}
    <PrototypeSwitcher variants={['A','B','C']} current={variant} />
  </>
);
```

对于子形态 A（现有页面）：在切换组件上方保留所有现有的数据获取逻辑；仅渲染树的部分会随变体切换而变化。

对于子形态 B（新页面）：位于 `/prototype/<name>` 的临时路由同样使用上述切换组件。

### 4. 构建浮动切换器

屏幕底部中央放置一个固定定位的小型工具栏，包含三部分内容：

- **左箭头**——循环切换到上一个变体（循环回第一项）。
- **变体标签**——显示当前变体的键值，若变体导出了名称，则同时显示名称。例如：“B — 侧边栏布局”。
- **右箭头**——循环切换到下一个变体（循环回第一项）。

行为说明：

- 点击箭头会更新 URL 查询参数（使用框架提供的路由方法——Next.js 中的 `router.replace`，React Router 中的 `navigate` 等），使变体状态可分享且在页面刷新后保持一致。
- 键盘操作：`←` 和 `→` 方向键也可用于切换。当 `<input>`、`<textarea>` 或 `[contenteditable]` 获得焦点时，不拦截方向键事件。
- 在视觉上与页面明显区分（如高对比度的圆角按钮、细微阴影），以明确表示其并非正在评估的设计的一部分。
- 生产环境构建时隐藏——通过 `process.env.NODE_ENV !== 'production'` 或类似检查进行条件编译，防止意外合并导致切换器被发布给用户。

将切换器封装成一个共享组件，以便两种子形态均可复用。将其放置在项目中共享 UI 组件的通用位置。

### 5. 交付成果

公开 URL（以及 `?variant=` 参数键值）。用户到达该页面时会逐一浏览各个变体。通常最有价值的反馈是：“我想要 B 的页头搭配 C 的侧边栏”——这才是他们真正想要的设计。

### 6. 记录结果并清理

一旦某个变体胜出，记录下最终选择及其理由，然后按照 [SKILL](SKILL.md) 中描述的方式归档该原型。将胜出的变体合并到正式代码中，其余部分则移至废弃分支，而非合并到主分支：

- **子形态 A**——将胜出的变体合并到现有页面；将失败的变体及切换组件从主分支移除。
- **子形态 B**——将胜出的变体提升为正式路由；将临时路由及切换组件从主分支移除。

由于完整的变体集合才是原始来源，因此应将其保留在废弃分支中，而不是直接丢弃——留在主分支的变体组件和切换器会迅速腐化，并让后续维护者感到困惑。

## 反模式

- **仅在颜色或文案上有所区别的变体。** 那只是微调，不是真正的原型。真正的变体应在结构上存在分歧。
- **变体之间共享过多代码。** 共享 `<Header>` 没问题；但共享 `<Layout>` 则违背了原型设计的初衷。每个变体都应能自由地舍弃现有布局。
- **将变体与真实的数据变更关联起来。** 只读的原型是可以的。如果某个变体需要触发变更，请指向一个模拟接口——问题在于“它应该是什么样子”，而非“后端是否正常工作”。
- **直接将原型推送到生产环境。** 变体代码是在原型开发约束下编写的（无测试、极少错误处理）。在正式合并时，请务必重新编写。
