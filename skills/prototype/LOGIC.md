# 逻辑原型

一个小型的交互式终端应用，允许用户手动驱动状态模型。当问题涉及**业务逻辑、状态转换或数据结构**时使用——这类问题在纸上看起来合理，但只有在通过实际案例验证后才会发现其中的问题。

## 何时适合这种形式

- “我不确定这个状态机是否能处理 X 后再 Y 的边界情况。”
- “这个数据模型是否真的能够表示……的情况？”
- “我想在编写 API 之前先感受一下它的设计应该是什么样。”
- 任何用户希望**按按钮并观察状态变化**的场景。

如果问题是“它应该是什么样子”——那就走错了分支。请使用 [UI.md](UI.md)。

## 流程

### 1. 明确问题

在编写代码之前，先写下你要进行原型设计的状态模型和具体问题。用一段话写在原型的 README 文件中，或者放在文件顶部的注释里。回答错误问题的逻辑原型纯属浪费——把问题明确下来，以便日后检查，无论是当前观看者还是离开后回来的人。

### 2. 选择语言

使用宿主项目所用的语言。如果项目没有明显的运行环境（例如文档仓库），则询问相关人员。

遵循项目现有的工具链约定——不要为了原型而引入新的包管理器或运行时环境。

### 3. 将逻辑隔离到一个可移植的模块中

将真正解决问题的逻辑部分封装在一个小而纯粹的接口背后，这个接口将来可以被提取出来并放入实际的代码库中。围绕它的 TUI 是一次性的；逻辑模块则不是。

合适的形态取决于问题：

- **纯 reducer** — `(state, action) => state`。适用于动作是离散事件且状态是一个单一值的情况。
- **状态机** — 显式定义状态与转换。适用于“当前哪些动作是合法的”本身就是问题的一部分时。
- **一组针对普通数据类型的纯函数**。适用于不存在隐式当前状态，只有变换操作的情况。
- **具有清晰方法接口的类或模块**，当逻辑确实需要维护内部状态时。

根据要解决的问题选择最合适的形态，而不是选择最容易与 TUI 搭配的形式。保持纯粹：不涉及 I/O、不包含终端代码、不使用 `console.log` 来控制流程。TUI 只负责导入并调用逻辑模块，逻辑模块不应向 TUI 传递任何信息。

这正是使原型在完成使命后仍具价值的原因：当问题得到解答后，经过验证的 reducer / 状态机 / 函数集合可以直接融入实际模块中。

### 4. 构建一个仅暴露状态的最小化 TUI

构建一个**轻量级的 TUI**——每一步都清屏（使用 `console.clear()` 或 `print("\033[2J\033[H")` 等等），然后重新渲染整个画面。用户应始终看到一个稳定的视图，而不是不断增长的滚动缓冲区。

每一帧分为两部分，顺序如下：

1. **当前状态**，以易于比较的方式呈现（每条字段占一行，或格式化为 JSON）。对字段名或章节标题使用**加粗**，对不太重要的上下文（如时间戳、ID、派生值）使用**浅色**。原生 ANSI 转义码即可——`\x1b[1m` 加粗，`\x1b[2m` 浅色，`\x1b[0m` 重置。除非项目中已有样式库，否则无需额外引入。
2. **键盘快捷键**，列于屏幕底部：`[a] 添加用户  [d] 删除用户  [t] 让时钟前进  [q] 退出`。对按键名称加粗，描述部分浅色，反之亦可——以阅读清晰为准。

行为：

1. **初始化状态**——一个内存中的对象或结构体。启动时渲染第一帧。
2. **每次读取一个按键（或一行输入）**，将其分发给一个修改状态的处理器。
3. **每执行完一个动作后重新渲染**整个画面——不追加内容，而是完全替换。
4. **循环直至退出**。

整个画面应能完整显示在单屏内。

### 5. 使其可通过一条命令运行

在项目的现有任务运行器中添加一个脚本（如 `package.json` scripts、`Makefile`、`justfile`、`pyproject.toml`）。用户只需运行 `pnpm run <prototype-name>` 或类似命令，无需记住路径。

如果宿主项目没有任务运行器，则直接将运行命令写在原型的 README 文件顶部。

### 6. 交付使用

将运行命令交给用户。他们会亲自操作，有趣之处往往在于他们说“等等，那不应该发生”或“咦，我还以为 X 会不一样呢”——这些正是隐藏在**想法**中的缺陷，而这正是原型的意义所在。如果用户要求新增动作，就予以实现。原型是会演化的。

### 7. 记录答案与原型本身

一旦原型解决了其问题，记录下答案，并按照 [SKILL](SKILL.md) 中所述的方式保存原型。逻辑层面的映射：经过验证的 reducer / 状态机 / 函数集合会被整合进实际模块（决策被吸收）；而 TUI 层则作为一次性组件，保留在废弃分支中，作为主要来源存档。

## 反模式

- **不要添加测试。** 需要测试的原型已经不再是原型了。
- **不要连接真实数据库。** 除非问题特别关注持久化，否则使用内存存储。
- **不要泛化。** 不要问“如果我们以后想支持 X 怎么办”。原型只解决一个问题。
- **不要让逻辑与 TUI 混淆。** 如果 reducer / 状态机 引用了 `console.log`、提示信息或终端转义码，则不再具备可移植性。TUI 应仅作为纯逻辑模块的薄层外壳。
- **不要将 TUI 层投入生产。** TUI 层专为终端手动操控优化，真正值得保留的是其背后的逻辑模块。
