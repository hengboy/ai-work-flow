# Git Committer 授权范围实施计划

## 目标

允许 Git Committer 在首次因工作区存在无关变更或提交范围不清而停止后，接受用户对首次阻塞时所列全部文件的一次性明确授权，并在第二次提交尝试中仅将该固定清单作为允许范围。任何授权后新增、首次未列出或未被用户明确授权的文件都不得进入提交，任何角色都不得依据相关性自行扩大范围。

## 需求

- 首次提交检查仍按严格白名单执行：文件必须是本次任务明确产生，或由用户逐一明确指定。
- 首次检查因发现不满足既有来源条件的变更或范围不清而停止时，Git Committer 必须输出当时全部待提交工作区变更的完整文件清单，形成后续唯一可引用的授权快照。
- 只有用户在该阻塞之后明确授权“将刚才列出的全部文件一并提交”或语义等价、且明确指向该完整清单的指令，Orchestrator 才能发起第二次 Git Committer 委派。
- 用户在首次阻塞前给出的“提交全部变动”等笼统指令不能预先授权未知清单；未明确指向刚才清单的笼统确认也不能扩大提交范围。
- 第二次委派必须携带首次清单的逐项、原样文件路径；Git Committer 不得根据对话、任务关系、文件名、diff 或其他推测重建或扩展清单。
- 授权清单只用于紧接该授权的一次提交尝试。尝试开始、范围校验失败、提交钩子失败或提交结束后均视为已消费；再次提交必须重新检查并取得新的明确授权。
- 当前工作区只要出现不在授权快照中的新增、修改、删除、重命名、复制或未跟踪文件，第二次尝试就必须停止，且不得暂存或提交任何文件。
- 如果授权快照中的文件在第二次检查时已经消失、路径变化或不再是待提交变更，视为工作区与授权快照不一致并停止，防止使用陈旧授权提交不同状态下的文件集合。
- 不引入基于“相关性”的判断，不将用户授权持久化到仓库或全局配置，不扩大 Git Committer 的 Git 操作权限。

## 实施决策

### 1. 定义首次阻塞清单协议

修改 `scripts/agent-assets/bodies/git-committer.md` 的工作边界和范围不清回复规则：

1. 在任何暂存动作之前读取完整 Git 状态，并以能够区分已跟踪修改、删除、重命名和未跟踪文件的状态结果构造快照；实现时优先使用可无歧义处理特殊路径的 porcelain `-z` 形式，不以自然语言解析 diff 推断路径。
2. 首次发现超出既有白名单的文件时立即停止，不调用 `git add`、`git commit` 或其他会改变状态的命令。
3. 阻塞回复保留明确的停止结论，同时逐项列出当次检查得到的完整文件清单。清单必须覆盖工作区中全部待提交路径，而不是只列出被判断为“无关”的部分。
4. 回复中声明该清单尚未获得授权，并要求用户明确引用“刚才列出的全部文件”；不得评价文件相关性或建议用户应授权哪些文件。
5. 对重命名或复制状态保留源路径和目标路径信息，使后续白名单可精确约束 Git pathspec；特殊字符路径必须使用一致、可比对的表示，避免显示值与第二次校验值不一致。

该修改替代当前 `scripts/agent-assets/bodies/git-committer.md:12` 和 `:16` 的“不得枚举”及“仅输出固定一句”约束，因为现有协议无法产生后续授权所需的固定清单。

### 2. 定义 Orchestrator 的跨轮授权传递

修改 `scripts/agent-assets/bodies/orchestrator.md`，新增 Git 提交授权专项规则：

1. 收到 Git Committer 的首次阻塞结果后，在当前对话状态中保留其完整清单，作为待授权快照；Orchestrator 不访问工作区，也不编辑、筛选或补全该清单。
2. 用户未明确授权刚才完整清单时，仅报告阻塞并等待，不得再次委派 Git Committer。
3. 用户明确授权后，第二次委派必须同时传入：用户的授权原文、首次清单的逐项路径、该清单为一次性白名单的声明，以及“发现任何当前变更集合差异即在暂存前停止”的要求。
4. 若当前上下文中无法恢复首次完整清单，或授权指向不唯一、清单被后续对话替换，则不得推断；重新委派 Git Committer 仅执行一次新的首次检查并生成新清单，旧授权不复用。
5. 第二次委派完成或失败后清除待授权快照；不得把它传给第三次尝试、其他任务或新会话。

这里的“跨轮”限定为同一 Orchestrator 对话上下文中的相邻用户轮次。仓库当前没有运行时授权状态存储，本需求不新增磁盘、Checkpoint 或全局配置持久化；新会话缺少原始清单时必须重新检查和授权。

### 3. 将共享约束加入生成源

修改 `scripts/agent-assets/routing.md`，在专项规则中加入对 Orchestrator 与 Git Committer 都生效的授权范围契约：

- 首次阻塞清单是唯一授权对象。
- 授权必须发生在清单展示之后并明确指向全部所列文件。
- Orchestrator 只能原样转交，Git Committer 只能逐项校验。
- 第二次检查的当前变更集合必须与授权快照一致；不一致时在暂存前停止。
- 白名单一次性消费，不能跨尝试、任务或新会话复用。
- 任一角色都不得推断文件相关性。

同时保持 `~/.config/ai-work-flow/routing.md` 仍由安装/生成流程从该资产生成，不直接修改用户全局安装结果。

### 4. 保持资产目录和角色配置稳定

检查但不修改 `scripts/agent-assets/roles.json` 与 `scripts/agent-assets/default-config.json`：

- `roles.json` 已允许 Orchestrator 委派 Git Committer，Git Committer 已拥有读取状态和执行受控 Git 命令所需的 `Read`/`Bash` 工具，不需要新增工具或委派能力。
- `default-config.json` 只定义平台模型配置，授权快照属于提示词协议而非新配置项，不应在环境配置中引入用户授权状态。
- 若实施时发现必须改变这两个文件才能满足生成器校验，应停止并重新确认设计，不能以顺带修改配置的方式扩大本计划。

## 接口与数据约束

### 授权快照

授权快照是一次首次阻塞结果中的完整文件集合。每项至少包含 Git 可用于第二次状态比较和精确暂存的路径；重命名/复制项必须同时保留源、目标关系。集合比较按规范化后的 Git 路径和状态进行，不按展示顺序、文件内容或业务相关性判断。

### 状态转换

1. `未检查`：尚无授权快照。
2. `已阻塞待授权`：首次检查停止并列出完整清单；未执行暂存。
3. `已明确授权`：用户明确指向上一状态的完整清单；Orchestrator 可进行一次第二次委派。
4. `校验中`：Git Committer 重新读取当前完整状态并与快照比较；任何差异都转为停止。
5. `已消费`：第二次尝试开始后，无论提交成功、钩子失败或范围校验失败，快照均不可再次使用。

### 提交前不变量

- 第二次检查得到的完整变更集合与授权快照逐项一致。
- 拟暂存路径等于授权快照，不是工作区全部路径的隐式通配。
- 暂存区不得包含授权快照以外的路径；如存在预先暂存且未授权的路径，必须停止，且不得通过 reset 等禁用操作改写暂存区。
- 提交信息仍只由 `git-commit` 技能产生，正常提交钩子仍执行。
- Git Committer 禁止修改文件、配置、分支、标签，禁止 stash、amend、push、reset、clean 或跳过钩子。

## 精确修改步骤

1. 在 `test/agent-workflow.test.mjs` 先增加失败断言，覆盖源提示词中的首次完整清单、事后明确授权、原样传递、集合一致性校验、一次性消费和禁止相关性推断。
2. 扩展同一测试中的安装/平台生成场景：执行现有 `install(paths)`，分别读取 Codex、Claude Code、OpenCode 生成的 `orchestrator` 与 `git-committer`，断言两类角色都包含各自应执行的授权契约；继续断言安装后的 `routing.md` 与资产源完全一致。
3. 修改 `scripts/agent-assets/bodies/git-committer.md`，落实首次阻塞清单格式和第二次白名单校验，移除与列清单相冲突的固定一句回复限制，同时保留既有 Git 操作白名单和 `git-commit` 技能门禁。
4. 修改 `scripts/agent-assets/bodies/orchestrator.md`，落实待授权快照的对话内保存、用户授权识别、第二次委派载荷及消费规则。
5. 修改 `scripts/agent-assets/routing.md`，写入共享的状态转换和禁止推断约束，确保安装到 `~/.config/ai-work-flow/routing.md` 后仍约束两个角色。
6. 运行定向测试并修正提示词或断言，直到源资产与三平台生成结果全部覆盖协议。
7. 运行完整测试，确认资产目录一致性、配置校验、安装幂等性、平台权限和现有规划/审查/重试规则没有回归。
8. 检查最终差异，确认仅包含上述三个提示词资产、`test/agent-workflow.test.mjs` 及本计划文件；`roles.json`、`default-config.json` 和任何生成到用户主目录的文件不应出现在差异中。

## 验证

### 自动化测试

1. 定向执行新增用例：

   ```bash
   node --test --test-name-pattern="Git Committer.*授权|authorized scope" test/agent-workflow.test.mjs
   ```

   用例名实施时统一选用一个稳定模式，确保命令实际命中新用例而不是零测试通过。

2. 执行生成链测试文件：

   ```bash
   node --test test/agent-workflow.test.mjs
   ```

3. 执行仓库完整测试：

   ```bash
   npm test
   ```

4. 执行差异检查：

   ```bash
   git diff --check
   git diff --name-only
   ```

### 测试场景

- 首次存在范围不清变更：停止且列出全部待提交文件，未暂存任何文件。
- 用户未明确引用刚才清单：Orchestrator 不发起第二次委派。
- 用户在清单出现前笼统说“提交全部变动”：不能视为后续授权。
- 用户在清单出现后明确授权全部所列文件：Orchestrator 原样传递清单，Git Committer 可进入一次范围复核。
- 授权后出现一个新未跟踪或已跟踪变更：第二次提交在暂存前停止，新文件和原清单文件都不提交。
- 授权后原清单路径消失、重命名或状态集合改变：旧授权失效并停止。
- 当前状态与快照完全一致：只以逐项路径暂存清单内文件，使用 `git-commit` 技能生成的信息提交并运行钩子。
- 第二次提交或钩子失败后再次请求提交：旧授权不可复用，必须重新列清单并取得授权。
- 三个平台生成的 Git Committer 与 Orchestrator 均保留对应规则，安装后的共享 routing 与源一致。
- `roles.json` 和 `default-config.json` 仍通过资产目录校验且内容不变。

## 风险

- **自然语言授权歧义：** “都提交”可能指任务文件而非刚才清单。通过要求授权发生在清单之后且明确引用完整清单来收紧，不做语义补全。
- **对话上下文丢失：** Orchestrator 没有持久状态存储。缺少原始逐项清单时强制重新检查，不能从摘要或 Git 当前状态反推旧快照。
- **检查与暂存之间的竞态：** 工作区可能在第二次状态检查后、暂存前变化。提示词应要求在首次状态比较后按明确 pathspec 暂存，并在提交前再次检查暂存区和工作区没有清单外变化；发现竞态即停止，不尝试修复工作区。
- **预先暂存内容：** 用户或其他进程可能已暂存未授权文件。提交前不变量同时约束暂存区和工作区，避免只检查未暂存 diff 后误提交。
- **特殊路径和重命名表示：** 普通行文本解析可能错误拆分空格、引号或重命名路径。使用 porcelain `-z` 的状态来源，并在两轮使用同一规范化规则。
- **提示词测试只能验证契约存在：** 当前测试体系生成静态代理文件，不能直接证明模型运行时百分之百遵循协议。自动化测试应验证所有生成平台包含完整、无冲突的约束，验收时再以受控临时仓库场景做人工行为核对；该核对不得对当前仓库执行真实提交。
- **与现有固定阻塞回复冲突：** 必须同步修改 Git Committer 工作边界和回复格式，避免一处要求列清单、另一处仍禁止枚举。

## 验收标准

- 首次范围阻塞输出包含当时全部待提交文件的完整清单，并明确未暂存、未提交。
- 只有清单展示后的明确整单授权才能触发一次第二次委派；预先或含糊的“提交全部”无效。
- 第二次委派载荷包含首次清单原文，Git Committer 不依赖自身跨轮记忆或相关性推断。
- 当前变更集合与授权快照有任何路径或状态差异时，Git Committer 在 `git add` 前停止。
- 集合一致时，拟暂存和最终提交文件严格等于授权快照，且提交信息和钩子规则不变。
- 授权在一次第二次尝试后失效，不能跨失败重试、其他任务或新会话复用。
- Codex、Claude Code、OpenCode 生成结果及安装后的共享 routing 均包含对应约束。
- `node --test test/agent-workflow.test.mjs`、`npm test` 和 `git diff --check` 全部通过。
- 最终实现差异不包含 `scripts/agent-assets/roles.json`、`scripts/agent-assets/default-config.json`、用户全局代理文件或任何无关文件。

## 范围外

- 不实现跨进程、跨会话或磁盘持久化的授权存储。
- 不修改 Checkpoint format，不把提交授权写入 execution Checkpoint。
- 不改变 Orchestrator、Git Committer 的模型、工具权限、委派关系或 workspace 等级。
- 不让 Git Committer判断哪些文件与任务相关，也不允许其自动清理、重置、贮藏或拆分工作区变更。
- 不新增提交、push、amend、跳过钩子或自动重试能力。
- 不直接生成或修改 `~/.config/ai-work-flow/`、`~/.codex/`、`~/.claude/`、`~/.config/opencode/` 下的实际安装文件。

## 假设

- 用户所说“跨轮传递”指同一 Orchestrator 会话中，首次阻塞结果与紧随其后的用户授权之间的对话状态传递；新会话不具备可靠的原始清单，因此必须重新检查。
- “刚才列出的全部文件一并提交”的语义要求当前变更集合与首次快照完全一致；如果原清单项消失或状态改变，宁可重新授权，也不使用部分或陈旧白名单。
- 现有安装器继续以 `scripts/agent-assets/` 为唯一生成源，测试中的临时 HOME/XDG_CONFIG_HOME 足以验证三平台产物而不污染用户真实全局配置。
